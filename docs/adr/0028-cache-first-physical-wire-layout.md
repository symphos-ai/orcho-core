# ADR 0028 — Cache-First Physical Wire Layout

Date: 2026-05-17
Status: accepted (in-progress; M10.5 lands in slices)
Supersedes: none (extends ADR 0026 §"Prefix vs Turn Payload")
Related: ADR 0009 (composable prompt parts), ADR 0026 (session-aware
prompt parts), ADR 0027 (execution surfaces / fanout reserved)

## Context

ADR 0026 introduced typed `PromptPart` metadata, the M2
`PromptRenderEnvelope` sidecar with `stable_prefix_parts` /
`turn_payload_parts`, a session-aware delta selector (M6), and trace
metadata (M7+). M11 closed the profile execution-policy knobs.

None of those milestones changed the **physical order of bytes** in
the wire prompt. After M11 the rendered prompt still glued the parts
as `prefix_upper + roles/<role> + tasks/<task> + formats/<format> +
extras + system_tail`. The role file embedded `$project_dir` and
`$context` substitutions, so byte 0 of every full prompt was
run-specific. Protected contracts (`plan_json`, `change_handoff`,
`authoring_language`, `plan_artifact_boundary`) sat at the end, after
dynamic task / artifact / critique text. The envelope metadata
described a stable prefix that the wire bytes did not actually
contain.

Provider prefix caching keys on leading bytes. Resumed-session delta
reuse (M6) still helped on long-lived sessions, but full / stateless
/ per_phase fresh-turn renders were getting approximately zero
prefix-cache hit. The cache-first thesis behind ADR 0026 was
unrealised in practice.

Related provenance bug: `PromptPart.source = "code-owned"` was being
applied to runtime-generated bodies (typed plan markdown injected
into implement, review critique injected into repair, contract
bundles). `code-owned` is meant for code/template constants; runtime
artifacts need their own provenance label.

## Decision

### Physical wire order

The wire prompt is assembled by cache breadth, broadest first:

```text
TIER GLOBAL                  (broadest cache; shared across runs/projects/users)
  universal protected strategies    — change_handoff, authoring_language
  phase output contracts            — plan_json, review_json, …
  role (default-shipped, no $vars)
  format (default-shipped, no $vars)
  task method (default-shipped, no $vars)

TIER WORKSPACE               (stable across runs in this workspace)
  role / format / task method overridden at workspace level
  workspace_context           (reserved; suppressed when empty)

TIER PROJECT                 (stable across runs targeting this project)
  role / format / task method overridden at project level
  project_context             — project_dir, project AGENTS.md, plugin config

TIER NONE                    (per-turn; never cached)
  task text
  artifact bodies
  reviewed file bodies
  critique / feedback / test output / diff
  round-specific data
```

The first dynamic byte appears only after every cacheable tier has
ended.

### Cache scope hierarchy

`PromptCacheScope` carries the full four-tier hierarchy:

| Scope | Stable while | Invalidates when |
|---|---|---|
| `GLOBAL` | Always, everywhere | orcho-core upgrade rewrites a shipped file |
| `WORKSPACE` | Pipeline targets the same workspace | workspace override edit; workspace switch |
| `PROJECT` | Pipeline targets the same project | project_dir change; project override edit; project AGENTS.md edit |
| `NONE` | Never | every call |

`SESSION` remains in the enum for legacy session-bound parts. `RUN`
stays available for any future part whose body legitimately varies
within one run identity; M10.5 does **not** use `RUN` for scope-stable
context, because `envelope.is_prefix_eligible()` excludes `RUN` from
`stable_prefix_parts` by design — relaxing it would dilute the RUN
meaning. Scope-stable context (project_context, workspace_context) is
`STATIC` + an explicit `cache_scope`; invalidation is body/hash-driven.

### Roles and task files are static

Roles describe persona only. Task files describe procedure only.
Neither contains `$<var>` placeholders. Runtime values arrive
through builders as typed dynamic parts (`turn_input`, `artifact`,
`feedback`), never via template substitution.

The discipline mirrors skill manifests: a skill does not embed its
own invocation input, and neither does a task method.

### Provenance labels

| Label | Meaning |
|---|---|
| `code-owned` | Body is a code / template constant. Protected contracts, minimal-intent fallback, the project_context layout (values embedded; layout itself code-defined). |
| `artifact` | Body was generated by an earlier agent or phase. Typed plan markdown, review critique, contract bundles, file-validation reviewed-file bodies, prompt prefixes assembled from prior phase outputs. |
| `core` / `workspace` / `project` | Body resolved from a `_prompts/...` markdown file via the override chain (default → workspace → project). |

### Override-aware classifier

The M3 classifier consults `prompt_loader.resolution_source()`:

```
if has_dollar_var(raw):
    return (TURN, NONE)
match resolution_source:
    case "core":      (STATIC, GLOBAL)
    case "workspace": (STATIC, WORKSPACE)
    case "project":   (STATIC, PROJECT)
```

A default-shipped role is `STATIC/GLOBAL`. A workspace override of
the same role is `STATIC/WORKSPACE` — it sits in the WORKSPACE tier
of the assembler output, after the GLOBAL tier. The cache-tier
hierarchy is uniform across part types.

## Cross-Run Cache Behavior

`orcho cross` walks multiple projects in one run. Provider prefix
cache view across calls in a single cross-run:

| # | Call | Hit | Miss |
|---|---|---|---|
| 1 | vue, plan round 1 | — (cold) | all |
| 2 | vue, plan round 2 | tiers GLOBAL + WORKSPACE + PROJECT(vue) | NONE |
| 3 | vue, validate_plan | same prefix; vue project_context still cached | NONE |
| 4 | **php**, plan round 1 | GLOBAL + WORKSPACE (project changes; vue's PROJECT bytes don't match) | PROJECT(php), NONE |
| 5 | php, plan round 2 | GLOBAL + WORKSPACE + PROJECT(php) | NONE |
| 6 | back to vue, cross_review | GLOBAL + WORKSPACE + PROJECT(vue) (provider still has it) | NONE |

PROJECT scope is the correct granularity for `project_context` in
both single-project and cross-project runs. The GLOBAL tier survives
project switches.

## Interaction with ADR 0027 (M13 fanout)

ADR 0027 introduces read-only execution surfaces ("fanout review"
lenses). Surfaces vary `PromptSpec` (often role + task) but share
base inputs and may share a physical session under
`session_split=common`. M10.5 must not bake assumptions that would
break that future shape.

Invariants M10.5 preserves:

- `make_session_key` does **not** include task name in scope (M11
  invariant).
- "STATIC stability" refers to a part's content being stable for a
  fixed `part_id` (`kind:name@version`). It does **not** mean
  "byte-identical across all `PromptSpec` variations." Different
  surfaces with different `PromptSpec.task` produce different
  `part_id`s and therefore different static parts; that is correct.
- Universal protected strategies (`change_handoff`,
  `authoring_language`) remain byte-identical regardless of surface.
  The cross-surface cache hit in fanout depends on this.
- Wire layout up to (but not including) the role part remains
  byte-identical across surfaces that vary only in role. Wire layout
  up to (but not including) the task-method part remains
  byte-identical across surfaces that vary only in task.

Forbidden in M10.5:

- No `$task_name`, `$phase_name`, `$surface_id` placeholders in
  protected contracts.
- No runtime composition logic that synthesises task-method from a
  template + task identifier. Task identity = file selection
  (`tasks/<name>.md`), not template substitution.
- No change to `make_session_key` shape, signature, or scope
  encoding.

## Consequences

### Positive

- Provider prefix cache reaches its design intent: thousands of bytes
  of universal contracts + role + format + task method + project
  context are reusable across runs.
- Cross-project runs preserve GLOBAL cache across project switches;
  only PROJECT bytes swap.
- Provenance trace tells the truth: `code-owned` means code-owned,
  `artifact` means runtime-generated, override-resolved bodies carry
  their resolution level as cache scope.
- The wire-layout test suite owns the invariant going forward; the
  bug class "metadata describes a prefix the wire bytes do not
  contain" is detectable.

### Negative / cost

- Wire bytes shift versus pre-M10.5. Existing prompt-shape snapshot
  tests need targeted regeneration; the wire-cache-layout test file
  is the new authority on the layout invariant.
- Task-method extraction may produce empty task files for tasks that
  were essentially `$task` plus trivial framing. Such files are
  deleted; the corresponding phase still runs (builder emits only
  the `turn_input` part).
- Authors of project / workspace prompt overrides can no longer rely
  on `$task` / `$project_dir` / `$context` substitution inside their
  overridden role or task files. Those values arrive as separate
  typed parts.

### Risks

| ID | Risk | Mitigation |
|---|---|---|
| R1 | Snapshot regen across many existing tests | Regenerate one builder at a time inside the step that shifts that builder; each regen reviewed individually |
| R2 | `authoring_language` directive previously sat next to task body by accident of glue order; moving to GLOBAL tier may regress body-language compliance | Smoke each affected mode under `--mock`; if compliance drops, classify the directive as a late-protected part inside GLOBAL tier (at end of tier, closer to dynamic input) |
| R3 | Workspace/project override classifier change surfaces previously-hidden override-resolution bugs | `tests/integration/test_prompt_resolution.py` is the safety net |
| R4 | Cross-project builders bypass `_render_prompt_output` and call `compose_prompt` directly | Step 5 audits and routes them through the cache-first gateway |
| R5 | Task-method extraction may empty a task file | Delete the file; let the builder emit only the `turn_input` part |
