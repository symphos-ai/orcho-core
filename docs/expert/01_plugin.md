# plugin.py — full reference

## Why a plugin

Without a plugin Orcho runs in generic mode. With a plugin it knows:
- What language/stack the project is written in
- Which files and directories are key
- How to run tests (via the ``tests`` quality gate)
- Extra instructions for each phase

## Location

```
your-project/
└── .orcho/multiagent/plugin.py
```

## All PluginConfig fields

Fields are defined in `pipeline/plugins.py::PluginConfig`. All are optional;
core falls back to generic defaults when a field is not set. The plugin exports
`PLUGIN: dict` with any subset of these fields — `load_plugin` filters
unknown keys and logs a warning instead of failing.

```python
from pipeline.plugins import PluginConfig

PLUGIN = dict(
    # ── Identity ────────────────────────────────────────────────
    name="My Project",              # Name (shown in logs)
    language="Python 3.12",         # Primary language / stack

    # ── Context for agents ──────────────────────────────────────
    architecture="""
        REST API. Controllers in src/api/, models in src/models/.
        Business logic in src/services/. Tests in tests/.
        Auth middleware in src/middleware/auth.py.
    """,

    # ── Important directories ───────────────────────────────────
    file_hints=[
        "src/api/",          # the agent looks here first
        "src/services/",
        "tests/",
    ],

    # ── Allowed companion modifications ─────────────────────────
    # Flat list of "glob — reason" entries naming satellite files whose
    # modification is NOT a scope violation in any task — lockfiles,
    # golden snapshots, regenerable artifacts. Review gates surface this
    # list so touching these files is not flagged as out-of-scope; the
    # content of the change is still reviewed normally. Informational
    # only — core does no glob matching or diff enforcement.
    allowed_modifications=[
        "package-lock.json — derived from package.json",
        "tests/golden/*.snap — golden snapshots regenerated by tests",
        ".orcho/runs/**/verify-report.json — verification report output",
    ],

    # ── Artifacts (where to put plans) ──────────────────────────
    ma_artifacts_dir=".orcho/docs",   # default: .orcho/artifacts

    # ── Extra per-phase instructions ────────────────────────────
    plan_prompt_extra="""
        Always check existing similar endpoints before creating new ones.
        Keep controllers thin — business logic goes to services.
    """,
    build_prompt_extra="""
        Run: composer test  — after making changes.
        Follow PSR-12 coding standard.
    """,
    review_focus_extra="""
        Check: N+1 queries, missing indexes, unvalidated inputs.
    """,

    # ── Testing (Phase 5e: declarative gate config) ─────────────
    # The former ``testing: TestingConfig`` field is gone — the config
    # moved inside ``quality_gates["tests"]``. The built-in ``tests``
    # gate parses this dict into ``TestingConfig`` inside the
    # pipeline; custom plugins declare the config with the same
    # schema without constructing ``TestingConfig`` directly.
    quality_gates={
        "tests": {
            "run_command": "pytest tests/ -x -q",
            "fail_keyword": "FAILED",
            "timeout": 120,
        },
    },

    # Runtime selection is not configured in PluginConfig. Use
    # _config/config.local.json phases.<phase>.provider/model instead.
)
```

## Minimal plugin (real example)

```python
PLUGIN = dict(
    name="Magica API",
    language="PHP 8.3",
    architecture="Custom MVC. Controllers: src/Controller/, Models: src/Model/",
    file_hints=["src/Controller/", "src/Model/", "src/Service/"],
    build_prompt_extra="Run: vendor/bin/behat after changes.",
    review_focus_extra="Check SQL injection, PDO param binding, missing migrations.",
)
```

## Nested git repository (monorepo / nested git)

If the project's git repository is **not at the root** of the project folder
(for example `my-mono/SubProject/.git`), set `git_dir` in the
**workspace config** — not in `plugin.py`:

```bash
# During interactive init Orcho finds the nested .git itself and asks.
orcho workspace init ~/www/my-org

# Or set it manually in .orcho/config.local.json:
```

```json
{
  "projects": {
    "my-mono": {"path": "/path/to/my-mono", "git_dir": "SubProject"}
  }
}
```

Orcho uses this path as the git root for worktree isolation and diff capture
(see ADR 0062). The `git_dir` field in `plugin.py` was removed in ADR 0062.

## Normalization on load

`load_plugin` accepts any dict under the `PLUGIN` key. Behavior:

- Fields missing from `PluginConfig.__dataclass_fields__` are ignored and
  logged as a warning (`plugin: ignoring unknown PLUGIN keys: ...`).
- `artifacts` accepts either `ArtifactsConfig` or a `dict`; the dict is coerced
  with `profile` validated against `ARTIFACT_PROFILES`.
- `allowed_modifications` must be a `list[str]`. A non-list value is dropped
  entirely (empty-list default); a list keeps only its string elements and
  drops non-strings. Both cases log a warning. The loader never raises.

This is a normalizing validator, not a strict one — it deliberately drops
unknown fields so the plugin does not fail because of a stale key after a
core upgrade. If you need strict validation, check PLUGIN before exporting
it inside the plugin itself.

## Workspace-level prompts

If several projects share one workspace, prompts can be overridden for all of them:

```
workspace-orchestrator/
└── .orcho/multiagent/prompts/
    └── tasks/build.md   ← applies to every project in the workspace
```
